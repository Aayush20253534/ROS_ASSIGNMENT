# `ros2_ws/src/` — Package Layout

This folder holds the ROS 2 packages. Each subfolder with a `package.xml` is a
**package**. `colcon build` (run from `ros2_ws/`) discovers and builds them all.

```
src/
├── TOPICS.md            <- list of every topic (READ THIS FIRST)
│
├── gui_interface/       <- SHARED library: defines all topic names in one file
│   └── gui_interface/topics.py
│
├── button_listener_pkg/ <- PROVIDED EXAMPLE: a node that LISTENS (GUI -> ROS)
│   └── button_listener_pkg/button_listener.py
│
└── student_nodes_pkg/   <- YOUR PACKAGE: add your own nodes here
    └── student_nodes_pkg/
        ├── example_publisher.py       (example: ROS -> GUI)
        ├── temperature_publisher.py   (fake Float32 sensor)
        └── temperature_recorder.py    (command-controlled CSV writer)
```

## Which file does what

| File / folder | Role | Do students edit it? |
|---------------|------|----------------------|
| `gui_interface/gui_interface/topics.py` | One place to define every topic name | Yes — add new topic constants |
| `student_nodes_pkg/student_nodes_pkg/*.py` | Your node code | Yes — add `.py` node files |
| `student_nodes_pkg/setup.py` | Registers your nodes for `ros2 run` | Yes — add a `console_scripts` line per node |
| `student_nodes_pkg/package.xml` | Declares dependencies | Sometimes — add `<exec_depend>` for new message types |
| `button_listener_pkg/...` | Worked example to copy from | Read-only reference |
| `TOPICS.md` | Human list of all topics | Yes — add a row per new topic |
| `../data/temperature_readings.csv` | Recorder output generated at runtime | No — inspect or export it |
| `build/`, `install/`, `log/` (in `ros2_ws/`) | Auto-generated by `colcon build` | Never — do not edit |

## The 4-step recipe to add a node

1. Create `student_nodes_pkg/student_nodes_pkg/my_node.py` (copy `example_publisher.py`).
2. Add the topic name to `gui_interface/gui_interface/topics.py` and document it in `TOPICS.md`.
3. Register it in `student_nodes_pkg/setup.py` under `console_scripts`.
4. Rebuild and run:
   ```bash
   colcon build          # run inside ros2_ws/
   source install/setup.bash
   ros2 run student_nodes_pkg my_node
   ```

Because rosbridge exposes every topic to the browser, the React GUI can reach
**any** topic you create — just subscribe/publish to the same name there.

## Running the included sensor and recorder manually

The normal Docker and Codespaces startup paths run these automatically. To run
them yourself after building and sourcing the workspace:

```bash
ros2 run student_nodes_pkg temperature_publisher
ros2 run student_nodes_pkg temperature_recorder
```

The GUI publishes `start`, `stop`, and `reset` on `/recorder/command`. Recorded
rows are written to `ros2_ws/data/temperature_readings.csv`.

## Adding multiple nodes in one package

A package can hold as many nodes as you like. Each node = **one `.py` file** +
**one line** in that package's `setup.py`.

1. Create the node file, e.g. `student_nodes_pkg/student_nodes_pkg/temperature_publisher.py`.
2. Register it in `student_nodes_pkg/setup.py` under `console_scripts` — one
   line per node:
   ```python
   entry_points={
       'console_scripts': [
           'example_publisher = student_nodes_pkg.example_publisher:main',
           'temperature_publisher = student_nodes_pkg.temperature_publisher:main',  # new
           'motor_controller = student_nodes_pkg.motor_controller:main',           # another
       ],
   },
   ```
   The format is always:
   ```
   'name_you_type = package_name.file_name:function_name'
   ```
   - `name_you_type` -> what goes after `ros2 run student_nodes_pkg ...`
   - `file_name`     -> the `.py` file (no extension)
   - `function_name` -> almost always `main`
3. Rebuild (see below) and run: `ros2 run student_nodes_pkg temperature_publisher`.

## Do I need to rebuild? (Docker)

The Docker container runs `colcon build --symlink-install` **once, when it
starts** (see `docker-compose.yml`). After that, nothing rebuilds
automatically. Whether you need to rebuild depends on what you changed:

| What you did | Rebuild? | Why |
|--------------|----------|-----|
| Added a **new node** (new file + new `console_scripts` line) | **Yes** | `ros2 run` executables are generated from `setup.py` at build time. |
| Edited an **existing** node's `.py` code | No | `--symlink-install` symlinks Python files — just restart the node. |
| Changed `package.xml` dependencies | **Yes** | Dependencies are resolved at build time. |
| Added a topic constant in `topics.py` and imported it | No | Symlinked; picked up on node restart. |

### How to rebuild after adding a new node

**Option 1 — restart the container** (re-runs the startup build command):
```bash
docker compose restart ros2
# or
docker compose up
```

**Option 2 — rebuild inside the running container** (faster, no restart):
```bash
docker exec -it ros2_react_starter_ros bash
cd /ros2_ws
colcon build --symlink-install
source install/setup.bash
ros2 run student_nodes_pkg my_new_node
```

**Bottom line:**
- Editing existing node code -> just restart that node, no build.
- Adding a brand-new node -> rebuild (restart the container or run `colcon build` inside it).
